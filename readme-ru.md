# Архитектурная трансформация платформы «GoFuture» (Taxi Aggregator)

Проект перехода от монолитной архитектуры к микросервисной Event-Driven системе для обеспечения глобальной экспансии компании.

## 🎯 Бизнес-цели и результаты (Business Impact)
Метрики:
* **Time-to-Market:** Сокращение времени выхода новых функций с 6 месяцев до **2 недель**.
* **Масштабируемость:** Обеспечение обработки более **500 000 конкурентных поездок**.
* **Доступность:** Гарантированная доступность критических сервисов на уровне **99.95%**.
* **Надежность:** Снижение среднего времени восстановления (MTTR) с более чем 4 часов до **1 часа**.
* **Эффективность:** Снижение затрат на ресурсы на **40%** через внедрение автоскейлинга.

Архитектура:
* Полностью микросервисная архитектура.
* Развертывание в более чем 3 географических регионах.
* ML для динамического ценообразования и антифрода.
* SaaS (Multi-tenancy).

## 🛠 Ключевые архитектурные решения

### Специфика проекта
| **Критерий**                   | **Анализ**                                                                                                                                                                                                                                                                  |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Тип и Свойства процесса**    | Смешанный тип:<br>- **OLTP-транзакции** (бронирование поездок) , <br>- **Real-time стриминг** (геолокация водителей, динамические цены) ,<br>- **Batch аналитика**<br>- Процесс содержит сложные ветвления и высокую степень асинхронности (Celery, RabbitMQ).                          |
| **Масштаб и Нагрузка (Scale)** | **Highload**. <br>- Переход от 100 тыс. к 500 тыс. конкурентных поездок. <br>- Объем данных — **порядок Тб** (учитывая логи, историю поездок и ML-пайплайны).<br>- Требования к Latency — критически низкие для матчинга водителя и пассажира .                                                                      |
| **Критичность данных**         | **Availability > Consistency** (для Discovery/Booking) и **Consistency** (для Payments).<br>- Для системы заказа такси доступность (возможность уехать) критичнее мгновенной согласованности баланса во всех регионах.<br>Однако: <br>  - для платежей требуется **ACID**, <br>  - а для статусов поездки допустима **Eventual Consistency**. |
| **Сложность логики**           | **«Ювелирная» обработка**. <br>- динамическое ценообразование в реальном времени ,<br>- «умное» перераспределение водителей (предотвращение «горячих зон»)<br>- ML-антифрод.                                                                                                                                 |
| **Жизненный цикл и Состояние** | - необходимо помнить состояние каждой активной поездки (Restartability)<br>- и хранить историю для аналитики и регуляторов .                                                                                                                                                                |
| **Инфраструктурный контекст**  | Переход от **Monolith (Django/Python)** к **Microservices**. <br>Облако: **Yandex Cloud**. <br>Текущая контейнеризация — Docker.                                                                                                                                                 |
### Этапы трансформации проекта
1. **Decomposition & Integration:** 
    для снижения связности и повышения гибкости к запросам бизнеса
    - Распил монолита на микросервисы (Task 1) 
    - Внедрение Event-Driven Communication (Task 2).
    
2. **Global Infrastructure:** 
    для обеспечения доступности 99,95% и низкой latency
    - Настройка Geo-distribution
    - и Failover в >3 регионах (Task 3).
    
3. **Data & Intelligence:**
    для ювелирной реакции на изменения в реальном времени
    - Построение пайплайна сбора данных для ML-моделей прогнозирования спроса (Task 4).
    
4. **Platformization:**
    для быстрой экспансии партнеров
    - Реализация мультитенантности 
    - и IAM для безопасного подключения партнеров (Task 5).

### 1. Декомпозиция монолита (Task 1)
Разработка стратегии распила монолитной кодовой базы на независимые доменные сервисы.
- Спроектировала карту микросервисов с четкими доменными границами (**DDD**).
- Разработала поэтапный план миграции на базе паттернов **Strangler Fig**, **CDC** и **Kafka Connect** для обеспечения консистентности данных без остановки бизнес-процессов.
- Внедрила **Anti-Corruption Layer (ACL)** для безопасной интеграции новых сервисов с легаси-системой.

### 2. Event-Driven Architecture & Consistency (Task 2)
Проектирование событийной платформы для обработки и мониторинга реалтайм потоков данных.
- Спроектировала карту доменных событий для обработки поездок (**Kafka**).
- Спроектировала распределенные транзакции через **Saga Orchestration** (**Temporal**). 
- Обеспечила согласованность через **Transactional Outbox** и **Idempotency** и отказоустойчивость через **Retries & Circuit Breaker**, 
- Спроектировала **Full-stack Observability** (**OTel, Prometheus, Grafana, Jaeger**)

### 3. Global Scaling & Security (Task 3)
Дизайн отказоустойчивой инфраструктуры в 3+ географических регионах.
- Разработала дизайн Multi-region развертывания (3+ региона) с учетом **Compliance** и использованием **GeoDNS, Healthchecks** и **Active-Passive replication**.
- Подготовила план аварийного переключения (**Failover**) для достижения доступности 99.95%.
- Выстроила эшелонированную защиту: **WAF, Regional Firewall, API Gateway, Transit Firewall**.

### 4. Data Pipeline & ML (Task 4)
Создание конвейера для работы ML-моделей динамического ценообразования и антифрода.
- Спроектировала Real-time конвейер обработки данных для ML-моделей динамического ценообразования и антифрода (**ETL, MLflow**). 
- Внедрила инструменты контроля качества данных (**Great Expectations**).

### 5. Multi-tenancy & B2B Ecosystem (Task 5)
Трансформация «GoFuture» в мультитенантную платформу для партнеров.
- Реализовала архитектуру мультитенантной платформы с изоляцией данных и интеграцией IAM (SSO)
- Автоматизировала процесс онбординга партнеров для сокращения времени запуска с 6 месяцев до 2 недель.

## 📚 Методология проектирования
Проект реализован с использованием архитектурных инструментов:
* **C4 Model:** Визуализация системы на уровнях Context (C1), Container (C2) и Component (C3).
* **Documentation as Code:** Использование нотаций Structurizr DSL и PlantUML.
* **Architecture Decision Records (ADR):** Ведение реестра решений с анализом компромиссов
