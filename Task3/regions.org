** Контекст
Для пользователей в Юго-Восточной Азии и Южной Америке требуется обеспечить 
- 99.99% доступности
- и низкую задержку (RTT < 100 мс для SEA, < 50 мс для SA)
- соответствие регуляторным требованиям (LGPD, PDPA, законы Индонезии и Вьетнама)

** Проблема
1. **Регуляторные требования** — разные страны требуют локального хранения персональных данных (PII). Нельзя обслуживать Индонезию из Сингапура, если индонезийский закон запрещает вывоз данных.
2. **Задержка** — один регион на всю Южную Америку даст высокую задержку для пользователей в Чили и Аргентине (данные идут через Бразилию).
3. **Доступность** — при отказе региона нужно переключить пользователей на резерв, но комплаенс может это запретить.

Выбрать регионы для развертывания для пользователей в Юго-Восточной Азии и Южной Америке.

** Решение
1. **Юрисдикционные регионы** — для стран с жёсткими законами (Бразилия, Индонезия, Аргентина)
2. **Регионы присутствия** — для стран с мягкими требованиями (Малайзия, Таиланд, Филиппины, Вьетнам с соглашением)
3. **Резервные регионы** — географически близкие, чтобы задержка при failover оставалась низкой

Для обеспечения 99.99% доступности
- минимум 2 зоны доступности на регион
    - При отказе основного региона пользователи переключаются на резервный (географически близкий)
- Ручной failover (автоматический рискован из-за split-brain)

Для обеспечения низкой задержки
- Пользователи направляются в ближайший регион по GeoDNS
- Резервные регионы выбираются с RTT < 100 мс от основного

** Принципы работы с PII **
| Принцип                                   | Что значит                                                                                              |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| PII не реплицируется между регионами      | Имя, телефон, адрес, платёжные данные не копируются в другой регион                                     |
| Каждый регион хранит свои данные локально | Индонезийские пользователи → данные в Джакарте                                                          |
| Путешествующие пользователи               | Writes → home-регион (где зарегистрирован пользователь), Reads → local cache (кеш в регионе пребывания) |

| ---------------------- | ------------------------------------ | ---------------------------------------- |
| Задержка для Индонезии | < 50 мс                              | < 20 мс                                  |
| Комплаенс Индонезии    | ❌ Нарушение (данные покидают страну) | ✅ Соблюдение                             |
| Сложность              | Низкая                               | Средняя                                  |

*** Почему не один регион на Южную Америку?
| Фактор                 | Один регион (Сан-Паулу) | Два региона (+ Буэнос-Айрес) |
| ---------------------- | ----------------------- | ---------------------------- |
| Задержка для Аргентины | 50-80 мс                | < 20 мс                      |
| Комплаенс Аргентины    | ⚠️ Соглашение           | ✅ PDPA (локальное хранение)  |
| Задержка для Чили      | 80-120 мс               | < 40 мс                      |


*** Регуляторные требования
| Страна        | Что разрешено                                                      |
| ------------- | ------------------------------------------------------------------ |
| **Индонезия** | Данные индонезийцев должны храниться в Индонезии (закон 2022 года) |
| **Вьетнам**   | Требует локального хранения данных (Cybersecurity Law)             |
| **Таиланд**   | Мягче, но рекомендует локальное хранение                           |
| **Бразилия**  | LGPD — жёстко требует хранения в Бразилии                          |
| **Аргентина** | Закон о защите данных (PDPA) — требует локального хранения         |

** Ограничения и риски
| Риск                                                                         | Вероятность | Смягчение                                                 |
| ---------------------------------------------------------------------------- | ----------- | --------------------------------------------------------- |
| **Путешествующий пользователь из Бразилии в Индонезию** не видит свои заказы | Средняя     | Кеш региона пребывания + асинхронная репликация (без PII) |
| **Задержка при failover** (Сингапур → Джакарта)                              | Низкая      | RTT 30-50 мс — приемлемо                                  |
| **Сложность синхронизации данных между юрисдикционными регионами**           | Средняя     | Только обезличенные данные (агрегаты, не PII)             |
| **Ручной failover дольше автоматического**                                   | Средняя     | RTO < 15 минут — допустимо для 99.99%                     |

** Альтернативы
| Альтернатива                                     | Почему отклонена                                                    |
| ------------------------------------------------ | ------------------------------------------------------------------- |
| **Физические регионы для каждой страны**         | Нереалистично: 10+ регионов, огромные затраты, сложность управления |
| **Один регион на весь мир (Европа)**             | Высокая задержка (300+ мс), нарушение комплаенса                    |
| **Только облачные регионы без учёта комплаенса** | Юридические риски, штрафы, блокировка сервиса                       |
| Меж-юрисдикционные развертывания                 | Невозможность реплицировать PII, возможны ограничения при создании заказа |









Для реализации агрегатора такси с доступностью 99.99% и низкой задержкой на двух континентах (ЮВА и Южная Америка), 
архитектура должна быть Multi-Region и Active-Active.

1. Архитектурная стратегия: «Edge Processing»
Чтобы обеспечить низкую задержку (Latency < 100ms) в Джакарте и Сан-Паулу одновременно, нельзя гонять данные в один центральный кластер.

2. Обеспечение доступности 99.99% (Four Nines)
Для достижения такого SLA обычного деплоя недостаточно:
Global Server Load Balancing (GSLB): Маршрутизация пользователя на ближайший живой регион.

Cluster Replication:
- Использование Confluent Cluster Linking или MirrorMaker 2 для синхронизации метаданных между регионами. 

Если регион в ЮВА «ляжет», данные о поездках уже должны быть (хотя бы с минимальной задержкой) в соседнем регионе.

Cell-based Architecture:
- Разделение трафика внутри региона на «соты» (например, по городам). Падение обработки в Сингапуре не должно затронуть Бангкок.

4. Критический нюанс: «Consistency vs Latency»
Exactly-once должен быть включен (processing.guarantee=exactly_once_v2), чтобы при переподключениях из-за плохого сигнала в джунглях или фавелах цена не пересчитывалась дважды, пугая клиента.

5. Инженерный компромисс (Trade-off)

При развертывании в >3 гео-зонах вы столкнетесь с проблемой Data Silos (фрагментация данных).
Главный риск: Синхронизация цен между регионами при переезде пользователя через границу.

Эшелонированную архитектуру (Tiered Architecture).
Когда мы говорим про 500к поездок в 3+ регионах, «центр» и «регион» — это не обязательно географические точки, это уровни принятия решений.

3. Где тут риск?
Если мы решим считать Surge во всего одном центральном Flink-кластере (например, только в США) для всего мира:
Latency: Водитель в Таиланде ждет цену лишние 300мс.
Availability: Упал канал связи через океан — весь Таиланд ездит по фиксированной цене, компания теряет миллионы на Surge.
