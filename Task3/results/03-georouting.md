# ADR-\_: Геомаршрутизация

## Контекст

В глобально распределённой системе с несколькими географическими регионами запросы пользователей должны направляться в тот регион, который обеспечивает минимальную задержку и соответствует регуляторным требованиям к хранению данных, без участия пользователя в выборе региона.

## Проблема

|  |  |  |
|----|----|----|
| Аспект | Вопрос | Почему это проблема |
| ———————- | ———————————————————————————————————————— | —————————————————————————————————- |
| ****Задержка**** | Как пользователь из Индонезии попадает в Джакарту (RTT ~20 мс), а не в Сингапур (~50 мс) или Сан-Паулу (~300 мс)? | Без геомаршрутизации пользователь может попасть в «неправильный» регион и получить высокую задержку |
| ****Комплаенс**** | Как гарантировать, что данные бразильского пользователя никогда не покинут Бразилию (LGPD)? | DNS или GeoIP могут ошибаться; пользователь может использовать VPN с IP другой страны |
| ****Отказоустойчивость**** | Как при отказе региона Сингапур автоматически перенаправить пользователей из Малайзии в Куала-Лумпур (резервный регион)? | Нужен механизм, который понимает, что регион недоступен, и переключает трафик |
| ****Прозрачность**** | Как сделать так, чтобы пользователь не выбирал регион вручную (как в настройках приложения)? | Технические пользователи могут справиться, но массовый пользователь ожидает, что «всё работает само» |

## Решение

- ****GeoDNS (AWS Route 53 / Cloudflare):**** Определяет регион пользователя по IP-адресу
- ****Latency-based routing:**** Направляет в регион с минимальной задержкой
- ****Health checks (глобальный сервис AWS Route 53):**** Исключает недоступные регионы из DNS-ответов
- ****Комплаенс-правила:**** Пользователи из Индонезии и Бразилии направляются только в свои юрисдикционные регионы

Схема маршрутизации

|  |  |  |  |
|----|----|----|----|
| Страна пользователя | Основной регион | Резервный регион | Причина |
| —————————- | ————— | —————- | —————————- |
| Индонезия | Джакарта | Сурабая | Локальный закон |
| Бразилия | Сан-Паулу | Рио-де-Жанейро | LGPD |
| Малайзия, Таиланд, Филиппины | Сингапур | Куала-Лумпур | Соглашение о передаче данных |
| Аргентина, Чили | Буэнос-Айрес | Кордова | PDPA |

****Проверка healthcheck:****

1.  Health Checker отправляет HTTPS-запрос на публичный эндпоинт API Gateway: GET /health
2.  API Gateway (внутри региона) проверяет доступность:
    - PostgreSQL (SELECT 1)
    - Kafka (metadata fetch)
    - Критические микросервисы (по необходимости)
3.  Если все проверки успешны, API Gateway возвращает HTTP 200
4.  При трёх последовательных неудачах регион исключается из DNS-ответов

## Аргументация

### Почему GeoDNS, а не Anycast

|  |  |  |
|----|----|----|
| Критерий | GeoDNS (Route 53) | Anycast (Cloudflare / AWS Global Accelerator) |
| ———————— | —————————————– | ——————————————— |
| Сложность настройки | Низкая | Средняя |
| Комплаенс-правила | ✅ Можно настроить явные правила для стран | ⚠️ Сложнее, требует BGP-политик |
| Health checks | ✅ Встроенные | ✅ Встроенные |
| Стоимость | Низкая | Высокая (фиксированный платёж за IP) |
| Точность гео-определения | Средняя (по IP) | Высокая (по BGP) |

Выбран GeoDNS,** потому что:

- Требует явных комплаенс-правил («только Индонезия → Джакарта»)
- Низкая стоимость при достаточной точности
- Простая интеграция с существующей инфраструктурой

### Почему не клиентская маршрутизация

- Пользователь не должен выбирать регион вручную
- Мобильные приложения не имеют встроенной GeoDNS-логики
- Обновление списка регионов потребовало бы обновления приложения

## Ограничения и риски

|  |  |  |
|----|----|----|
| Риск | Вероятность | Смягчение |
| ——————————————————————————- | ———– | ———————————————————————————– |
| DNS-маршрутизация не учитывает перегрузку регионов (только доступность) | Средняя | Дополнительный балансировщик на уровне API Gateway (по RPS) |
| Пользователи с VPN могут быть направлены в «неправильный» регион | Низкая | Принудительное перенаправление на уровне API Gateway по геолокации устройства (GPS) |
| При отказе резервного региона возможна деградация сервиса (повышенная задержка) | Средняя | Пользователь получает предупреждение о возможных задержках |
| Health check может не заметить частичный отказ (например, только БД) | Средняя | Составные health checks (API + БД + Kafka) |

## Альтернативы

|  |  |
|----|----|
| Альтернатива | Почему отклонена |
| ——————————————————————– | ———————————————————– |
| ****Anycast (AWS Global Accelerator)**** | Высокая стоимость, сложность настройки комплаенс-правил |
| ****Клиентская маршрутизация (приложение выбирает регион)**** | Требует обновления приложений, пользователь может ошибиться |
| ****Единый глобальный балансировщик (HTTP-редирект)**** | Добавляет дополнительный hop, увеличивает задержку |
| ****Ручное распределение (пользователь выбирает страну в настройках)**** | Плохой UX, не для массового пользователя |
