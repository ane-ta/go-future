# \[#A\] ADR-\_: План аварийного переключения

## Контекст

Два региона:

- Основной регион (Сингапур) — обрабатывает 100% трафика для SEA
- Резервный регион (Куала-Лумпур) — получает реплику данных, но микросервисы в режиме standby

При отказе активного региона нужно переключить трафик на резервный.

## Проблема

Как обнаружить отказ? Как переключить трафик? Как не допустить split-brain? Как вернуться на основной регион обратно?

## Решение

Ручной failover + ручной failback

1.  В DNS-записи изначально настроены оба региона через Weighted routing:
    - основной - 100 с HealthCheck
    - резервный - 0 без HealthCheck
2.  Детектирование отказа
    - Health Checker проверяет API Gateway каждого региона каждые 30 секунд
    - Health check считается проваленным после 3 последовательных неудачных попыток (90 секунд)
    - При провале health check'а
      - основной регион помечается Unhealthy,
      - отправляется алерт инженеру
      - dns возвращает резервный регион, как "режим последнего средства" (last resort)
        - пользователи начинают получать ошибки, так как микросервисы резервного региона в режиме standby
3.  Ручное переключение (failover) При отказе активного региона инженер:
    - получает алерт об отключении
    - Проверяет, что основной регион действительно недоступен (дашборд Grafana отсутствие метрик)
    - Проверяет, что данные в резервном регионе синхронизированы (лаг репликации = 0)
    - переключает DNS на резервный регион
    - Активирует микросервисы в резервном регионе (из standby в active)
      - Пользователи перестают получать ошибки
4.  Ручное восстановление (failback) После восстановления основного региона инженер:
    - запускает обратную репликацию (временные Debezium + JDBC Sink)
    - Дожидается синхронизации (лаг = 0)
    - Переключает резервный регион пассивный режим
      - Пользователи начинают получать ошибки
    - Проверяет синхронизацию (лаг = 0)
    - Останавливает обратную репликацию
    - Запускает прямую репликацию
    - переключает DNS на основной регион
      - Пользователи перестают получать ошибки

|  |  |  |
|----|----|----|
| Параметр | Значение | Обоснование |
| ——– | ———- | ————————————————————– |
| ****RTO**** | \< 15 минут | Ручной failover по утверждённому runbook |
| ****RPO**** | \< 5 минут | MirrorMaker 2 синхронизирует consumer offsets каждые 60 секунд |

## Аргументация

### Автоматический vs ручной failover

|  |  |  |
|----|----|----|
| Аспект | Автоматический | Ручной |
| ——————————- | —————————————————– | —————————– |
| ****Скорость**** | Секунды | Минуты (RTO 5-15 мин) |
| ****Риск ложного срабатывания**** | Высокий (может переключить при временном лаге) | Низкий (инженер проверяет) |
| ****Риск split-brain**** | Высокий (оба региона думают, что они активны) | Низкий (человек контролирует) |
| ****Сложность реализации**** | Высокая (нужны distributed consensus, lease, fencing) | Низкая (инженер по runbook) |
| ****Требования к инфраструктуре**** | Высокие (отдельный quorum, например, 3-й регион) | Низкие |

### Почему ручной failover:

1.  GoFuture не требует секундного восстановления.

- 99.99% доступности допускает ~52 минуты простоя в год. 15 минут на failover — это ~0.003% от года. Формально допустимом.
- RTO \< 15 минут. Бизнес готов подождать 15 минут, чтобы инженеры приняли решение.

Ручной failover:

1.  исключает ситуации Split-brain, которые могут спровоцировать конфликты и потерю данных

- Автоматический failover не может гарантировать, что старый мастер действительно остановился (проблема "fencing")

1.  исключает ложные срабатывания — временные сетевые проблемы или перегрузка не должны вызывать переключение региона

- Временная задержка сети или перегрузка API Gateway не означают отказ всего региона.
- Автоматический failover может переключить трафик при незначительной проблеме, создав ненужный инцидент.

Автоматический failover

1.  требует третьего региона или отдельной системы управления для кворума.
2.  избыточен по требованиям и сложен для поддержки

### Когда можно перейти на автоматический failover

Если в будущем понадобится RTO \< 1 минуты

## Ограничения и риски

|  |  |  |
|----|----|----|
| Риск | Вероятность | Смягчение |
| ————————————————————- | ———– | ——————————————————– |
| Человеческая ошибка при ручном failover | Средняя | Автоматизация runbook'а (Ansible), регулярные тренировки |
| Длительное окно недоступности при failback | Низкая | RTO \< 15 минут — бизнес допустимо |
| Рассинхронизация данных при сбое во время обратной репликации | Низкая | Идемпотентность consumer'ов, повторная синхронизация |

## Альтернативы

|  |  |
|----|----|
| Альтернатива | Почему отклонена |
| —————————————— | —————————————— |
| Автоматический failover (Active-Passive) | Риск split-brain, требует третьего региона |
| Active-Active с CRDTs | Высокая сложность, неоправданно для такси |
| Полное резервирование на уровне приложения | Дорого, избыточно для текущих требований |
