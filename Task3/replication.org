** Контекст
Для обеспечения доступности 99.99% и соблюдения регуляторных требований (LGPD, PDPA, локальные законы Индонезии) выбрана архитектура с двумя независимыми развертываниями в разных географических регионах:
- **Активный регион:** Сингапур (для SEA) / Сан-Паулу (для SA)
- **Пассивный регион:** Куала-Лумпур / Буэнос-Айрес

** Проблема
Необходимо спроектировать модель репликации данных между активным и пассивным регионами, которая обеспечивает:
- Синхронизацию операционных данных (PostgreSQL)
- Синхронизацию событий (Kafka) и consumer offsets
- Возможность восстановления (failback) без потери данных
- Соответствие регуляторным требованиям (PII не покидает юрисдикцию)

** Решение
*** Модель репликации: Active-Passive на CDC (Debezium + Kafka + MirrorMaker)

- **Активный регион:** 
  - Все микросервисы активны
  - PostgreSQL — источник правды
  - Debezium Source Connector читает WAL и публикует события в Kafka
  - Kafka служит шиной событий для EDA

- **Пассивный регион:**
  - Микросервисы в режиме standby (не обрабатывают запросы)
  - MirrorMaker 2 реплицирует топики и consumer offsets из активного Kafka
  - JDBC Sink Connector применяет изменения к пассивной PostgreSQL

| Компонент        | Технология                    | Роль                               |
| ---------------- | ----------------------------- | ---------------------------------- |
| CDC Source       | Debezium PostgreSQL Connector | Читает WAL → публикует в Kafka     |
| Репликация Kafka | MirrorMaker 2                 | Копирует топики + consumer offsets |
| CDC Sink         | Debezium JDBC Sink Connector  | Читает Kafka → пишет в PostgreSQL  |

*** Обратная репликация (failback)

При восстановлении основного региона:
- Запускается временный Debezium в пассивном регионе
- Запускается временный JDBC Sink в активном регионе
- После синхронизации трафик переключается обратно
- Временные коннекторы удаляются

Диаграмма - Region Deployment

** Аргументация
*** RPO / RTO

- **RPO (потеря данных):** < 5 минут (синхронизация offsets с интервалом 60 сек)
- **RTO (время восстановления):** < 15 минут (ручной failover)

*** Почему Active-Passive, а не Active-Active

- Active-Active требует сложного разрешения конфликтов (CRDTs / LWW)
- При Active-Active PII может непреднамеренно покинуть юрисдикцию
- Active-Passive проще в отладке и эксплуатации
- RPO < 5 минут и RTO < 15 минут соответствуют бизнес-требованиям

*** Почему MirrorMaker 2, а не другие инструменты

- Нативная поддержка в Kafka
- Синхронизация consumer offsets (ключевая функция для failover)
- Активно поддерживается и документирован

** Ограничения и риски
| Риск                                                 | Вероятность | Смягчение                                                    |
| ---------------------------------------------------- | ----------- | ------------------------------------------------------------ |
| **Дублирование событий при failover**                | Высокая     | Идемпотентность consumer'ов (обязательное требование к коду) |
| **Потеря данных при асинхронной репликации**         | Низкая      | RPO < 5 минут — бизнес приемлемо                             |
| **Человеческий фактор при ручном failover**          | Средняя     | Автоматизация runbook'ов (Ansible / Terraform)               |
| **Разрастание служебных топиков (mm2-offset-syncs)** | Низкая      | Настроена автоматическая очистка (retention 7 дней)          |
| **Регуляторные риски при failback**                  | Низкая      | PII реплицируется только внутри юрисдикции                   |
| **Увеличение сетевого трафика между регионами**      | Средняя     | Сжатие данных в Kafka, мониторинг bandwidth                  |

** Альтернативы
| Альтернатива                                                   | Почему отклонена                                                         |
| -------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Active-Active с CRDTs**                                      | Высокая сложность реализации, риск конфликтов                            |
| **Логическая репликация PostgreSQL (Publisher-Subscriber)**    | Не реплицирует Kafka events и consumer offsets                           |
| **Cloud-native DR (AWS Aurora Global Database / GCP Spanner)** | Вендор-лок, невозможность соблюсти комплаенс (данные не покидают регион) |
| **Полный Big Bang failover (без репликации)**                  | Недопустимый RPO (потеря всех данных за время отказа)                    |
| **Confluent Cluster Linking**                                  | Проприетарная технология, привязка к Confluent                           |
