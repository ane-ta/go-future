# ADR-\_: Внедрение ML-моделей

## Контекст

«GoFuture» нужно внедрение ML-моделей для максимально точного:

- динамического ценообразования,
- прогнозирования спроса
- и обнаружения мошенничества.

## Проблема

Спроектировать схему сбора данных из всех микросервисов с интеграцией в ML-платформу и BI-инструменты.

## Решение

Confluent + Great Expectations + MLflow (или Kubeflow)

Диаграмма

### Обеспечение качества данных

Трехслойная система контроля качества данных.

1.  Слой превентивного контроля (Инфраструктурный).
    - Техническая валидация (Data Contracts)
      - проверяет структуру сообщений
      - предотвращает падение Analytics Engine при нарушении формата данных
2.  Слой содержательного контроля (Аналитический)
    - Валидация бизнес-логики и аномалий (Statistical DQ)
      - проверяет правила бизнес-логики на сообщениях
      - предотвращает обучение моделей на логически ошибочных данных
3.  Слой статистического контроля (ML-специфичный)
    - Мониторинг «дрейфа» данных (Data Drift)
      - Сравнивает текущие статистические распределения с историческими в ML Platform.
      - Помогает выявлять аномалии переобучения ML модели
    - Обеспечение «свежести» данных (Freshness SLA)
      - Временные метки (timestamps) на каждом этапе обработки.
      - Гарантирует использование только актуальных данных для генерации предсказаний

### Инструменты:

- Confluent Schema Registry — контроль контрактов (формат).
- Great Expectations — контроль бизнес-правил (смысл).
- Prometheus & Grafana — оперативный мониторинг качества пайплайна (метрики задержек и ошибок валидации).

## Аргументация

Для работы ML-моделей (динамическое ценообразование, антифрод) критически важно качество данных, так как ошибки в них напрямую ведут к финансовым потерям.

## Ограничения и риски

- Увеличение задержки (Latency):
  - Дополнительные этапы валидации (Great Expectations) увеличивают время прохождения данных через пайплайн. Это критично для Real-time предиктов, поэтому проверки должны быть оптимизированы.
- Сложность поддержки:
  - Командам микросервисов теперь нужно поддерживать Data Contracts. Любое изменение в БД требует согласования со схемой в Registry.
- Ресурсные затраты:
  - Хранение дублирующих данных в DWH и Feature Store, а также вычислительные мощности для обучения моделей увеличивают облачный бюджет.
- Риск «отравления» моделей:
  - Несмотря на проверки, существует риск пропуска изощренных аномалий, что может привести к неверному ценообразованию.

## Альтернативы

Confluent + Great Expectations + MLflow (или Kubeflow) — это «золотая середина».

- Confluent уже внедрен (это экономит время и деньги).
- Great Expectations дает гибкость аналитикам.
- Open-source ML платформы позволяют не зависеть от капризов облачных провайдеров.

### Отсутствие слоя DQ (Data Quality):

Минус: Высокий риск финансовых потерь из-за некорректных предиктов. Стоимость исправления ошибки в данных после обучения модели в десятки раз выше, чем стоимость проверки на входе.

### Protobuf (gRPC) без Registry

Микросервисы обмениваются бинарными данными. Схема «зашита» в код (compiled). Минус: Аналитикам очень сложно вычитывать такие данные из Kafka, если у них нет актуальных .proto файлов.
