# ADR-\_: Переход на Event-Driven Architecture (EDA)

## Контекст

Для достижения масштабируемости в 500тыс одновременныъ поездок необходим переход на EDA

## Проблема

В текущей архитектуре:

- Синхронные вызовы между доменами создают жёсткие связи
- Отказ одного сервиса (например, Pricing) приводит к каскадному отказу всего заказа
- Масштабирование требует увеличения ресурсов всего монолита

Необходимо спроектировать доменные события и асинхронное взаимодействие для миграции на EDA.

## Решение

Диаграмма - ProdServices Logic Interactions

## Аргументация

### Почему оркестратор, а не хореография

- Ключевой бизнес-процесс (создание заказа) включает 5+ шагов с компенсациями
- При хореографии логика размазывается по сервисам, отладка становится сложной
- Оркестратор хранит состояние саги, что позволяет отслеживать статус заказа
- Оркестратор централизованно обрабатывает таймауты и компенсации

### Почему синхронные вызовы остались для fraud и payments.hold

- \`fraud.check()\` — быстрая операция (100 мс), пользователь ждёт результат
- \`payments.hold()\` — внешний API, пользователь ждёт подтверждения блокировки денег
- Асинхронная обработка этих шагов ухудшит UX (пользователь не увидит результат сразу)

### Почему остальные вызовы асинхронные

- Поиск водителя, уведомления, аналитика не требуют немедленного ответа
- Асинхронность позволяет разорвать жёсткие связи между сервисами
- При отказе сервиса уведомлений заказ всё равно будет создан

## Ограничения и риски

|  |  |  |
|----|----|----|
| Риск | Вероятность | Смягчение |
| ————————————- | ———– | —————————————————- |
| Потеря событий при падении брокера | Низкая | RabbitMQ Streams с репликацией, persistent messages |
| Дублирование событий | Средняя | Идемпотентность consumer'ов (проверка event<sub>id</sub>) |
| Задержки в доставке событий | Низкая | Мониторинг consumer lag, алерты при \>1000 сообщений |
| Сложность отладки распределённой саги | Средняя | Трассировка через OpenTelemetry, логирование saga<sub>id</sub> |
| Нарушение порядка событий | Низкая | Партиционирование по saga<sub>id</sub> гарантирует порядок |

## Альтернативы

|  |  |
|----|----|
| Альтернатива | Почему отклонена |
| —————————————————– | ———————————————————————————- |
| Чистая хореография (без оркестратора) | Логика размазывается по сервисам, сложно отлаживать, риск циклических зависимостей |
| Все вызовы синхронные (как в монолите) | Не масштабируется, каскадные отказы, высокое время ответа |
| Все вызовы асинхронные (включая fraud, payments.hold) | Плохой UX (пользователь не видит результат операции) |
| Использование Kafka вместо RabbitMQ Streams | Избыточно для текущих требований, но заложен путь миграции |
