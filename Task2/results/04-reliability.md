# ADR-\_: Обеспечить снижение инцидентов

## Контекст

В распределённой системе с 10+ микросервисами, асинхронными событиями и синхронными вызовами возникают новые типы отказов:

- Потеря событий при падении сервиса после коммита транзакции
- Дублирование событий при повторных доставках
- Каскадные отказы при недоступности внешних сервисов
- Перегрузка сервисов при резком росте запросов

Требуется обеспечить надёжность взаимодействий и снизить количество инцидентов.

## Проблема

|  |  |  |
|----|----|----|
| Проблема | Риск | Пример |
| ——————– | ————————– | ——————————————————— |
| Потеря события | Данные не синхронизированы | Заказ создан, но событие \`BookingCreated\` не опубликовано |
| Дублирование события | Повторная обработка | \`PaymentCharged\` дважды списал деньги |
| Каскадный отказ | Система падает целиком | Яндекс.Пэй тормозит → все сервисы ждут |
| Перегрузка сервиса | Отказ по ресурсам | 1000 запросов в секунду на сервис с пулом на 100 потоков |

## Решение

Использовать следующие механизмы надежности взаимодействий

|  |  |  |
|----|----|----|
| Механизм | Реализация | Где применяется |
| ——————————- | —————————————————————————————– | ———————————————- |
| ****Idempotency**** | \`event<sub>id</sub>\` (UUID) в каждом событии; consumer хранит \`processed<sub>events</sub>\`  (TTL = retention) | Все consumer'ы |
| ****Dead Letter Queue (DLQ)**** | Топик \`dead.letter.queue\` (1 партиция, retention 7 дней) | Все consumer'ы |
| ****At-least-once + idempotency**** | Брокер гарантирует доставку, consumer идемпотентен | Все топики (exactly-once не требуется) |
| ****Transactional Outbox**** | Таблица \`outbox<sub>events</sub>\` в каждой БД сервиса. CDC (Debezium) | Оркестратор (BookingCreated), Payments, Driver |
| ****Circuit Breaker**** | Отключает вызов после N ошибок за период, fallback через DLQ | Внешние API (Payments → Яндекс.Пэй, Geo → Яндекс.Карты) |
| ****Bulkhead**** | Ограничение параллельных запросов (thread pool) | Payments, Driver (защита от перегрузки) |
| ****Timeout**** | 5 секунд на вызов, при превышении → retry или отмена саги | Все синхронные вызовы оркестратора |

****Transcactional outbox****

- Каждый сервис имеет таблицу `outbox_events` в своей БД
- Бизнес-данные и событие сохраняются в одной транзакции
- CDC (Debezium) читает WAL PostgreSQL и публикует события в брокер

****Idempotency**** Каждое событие содержит `event_id` (UUID). Consumer хранит `processed_events` в Redis (TTL 7 дней). При повторной доставке проверяет `event_id` и игнорирует дубли.

****Retry, Cirquit breaker и DLQ****

- Экспоненциальная задержка: 1 → 2 → 4 → 8 секунд, затем Cirquit breaker и DLQ
- Dead Letter Queue: топик `dead.letter.queue`, мониторинг размера (\>100 → алерт)
- At-least-once + идемпотентность (exactly-once не требуется)

## Аргументация

### Почему Transactional Outbox, а не двухфазный коммит (2PC)

|  |  |  |
|----|----|----|
| Подход | Плюсы | Минусы |
| —————- | ——————————————————- | ——————————————————– |
| ****Outbox + CDC**** | Нет блокировок, масштабируется, нет единой точки отказа | Задержка (CDC), eventual consistency |
| ****2PC (XA)**** | Строгая согласованность | Блокировки, низкая производительность, не масштабируется |

Выбран Outbox, так как GoFuture требует высокой производительности и масштабируемости.

### Почему at-least-once + idempotency, а не exactly-once

|  |  |  |  |
|----|----|----|----|
| Подход | Сложность | Производительность | Гарантии |
| ——————————- | —————————————————— | —————— | —————————- |
| ****at-least-once + idempotency**** | Низкая | Высокая | Дубли возможны, но безопасны |
| ****exactly-once**** | Высокая (транзакционный producer, idempotent consumer) | Средняя | Нет дублей |

### Почему Circuit Breaker и Bulkhead

- Circuit Breaker: Защита от каскадных отказов. Если Яндекс.Пэй тормозит, не ждём 5 секунд каждый раз
- Bulkhead: Защита от перегрузки. Если на Payments пришло 1000 запросов/сек, не пускаем их все сразу

## Ограничения и риски

|  |  |  |
|----|----|----|
| Риск | Вероятность | Смягчение |
| ———————————————– | ———– | ————————————————————- |
| **CDC lag** (Debezium не успевает) | Низкая | Мониторинг лага, алерт \>1 сек |
| ****Переполнение DLQ**** | Низкая | Алерт при \>100 сообщений, автоматический ретрай |
| ****Потеря processed<sub>events</sub> в Redis**** | Низкая | TTL настроен, повторная обработка безопасна (идемпотентность) |
| ****Circuit Breaker отключает работающий сервис**** | Низкая | Настройка порогов, ручной override |
| ****Bulkhead очередь переполняется**** | Средняя | Мониторинг размера очереди, увеличение pool при необходимости |

## Альтернативы

|  |  |
|----|----|
| Альтернатива | Почему отклонена |
| ————————————– | ——————————————————– |
| ****Two-phase commit (XA)**** | Низкая производительность, блокировки, не масштабируется |
| ****Exactly-once с Kafka Transactions**** | Высокая сложность, требует координации producer/consumer |
| ****Outbox через опрос БД (polling)**** | Высокая нагрузка на БД, задержки |
| ****Sagas без компенсаций**** | Нет отката при ошибках |
| ****Без Circuit Breaker (только retry)**** | Риск каскадных отказов при зависании внешнего API |
