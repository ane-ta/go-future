# Архитектурный план:

## Транспорт:

- Kafka как единую шину (вместо RabbitMQ)
  - так как у Kafka лучше экосистема для Python-стриминга

Слой логики:

## Микросервисы

Синхронное взаимодействие

- FastAPI

Асинхронное взаимодействие

- питон-библиотека Faust (или его современный форк Faust-Streaming)
  - портирует концепции Kafka Streams на Python/Asyncio.
  - идеально для «прод-микросервисов» (прием заказов, смена статусов).
    - Команда пишет на привычном Python, используя async/await, и получает stateful-стриминг с локальным хранилищем (RocksDB).
- реестр схем данных событий
  - Confluent Schema Registry
    - Главная ценность — эволюция схем. Если новая версия схемы нарушает совместимость (например, удалили обязательное поле), то реестр запретит ее публикацию
- формат данных
  - Avro/Protobuf
    - Для 500к поездок это экономия терабайтов трафика в месяц.

## Иначе Python-команде будет тяжело с «чистым» стримингом

Для миграции с Django-монолита характерны 2 проблемы:

- Производительность:
  - Python медленнее Java на операциях сериализации/десериализации JSON/Protobuf в потоке. Вам придется агрессивно масштабировать количество подов в K8s.
- Типизация:
  - В стриминге ошибки в схеме данных «взрывают» пайплайн.
