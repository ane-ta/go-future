# ADR-\_: Схема топиков для EDA

## Контекст

Для перехода на Event-Driven Architecture (EDA) необходимо сконфигурировать брокер сообщений. От правильной конфигурации топиков, партиций и ключей зависит масштабируемость и производительность системы.

## Проблема

Необходимо:

- Определить количество партиций для каждого топика
- Выбрать ключи партиционирования для гарантии порядка событий
- Обеспечить региональную изоляцию данных
- Рассчитать пороги масштабирования при росте нагрузки

## Решение

### Схема топиков

1.  order.events

    - Количество партиций на регион: 2
    - Ключ партиционирования: region:saga<sub>id</sub>
    - Retention: 7 дней
    - Consumer capacity: 100 RPS
    - Обоснование: 500k конкурентных поездок дают 185 RPS на регион. 2 партиции дают предел 200 RPS. Запас 15 RPS.

2.  driver.location

    - Количество партиций на регион: 7
    - Ключ партиционирования: region:driver<sub>id</sub>
    - Retention: 1 день
    - Consumer capacity: 10 000 RPS
    - Обоснование: 1M водителей онлайн дают 70 000 RPS на регион. 7 партиций × 10 000 = 70 000 RPS ровно под нагрузку.

3.  driver.events

    - Количество партиций на регион: 2
    - Ключ партиционирования: region:saga<sub>id</sub>
    - Retention: 7 дней
    - Consumer capacity: 100 RPS
    - Обоснование: аналогично order.events

4.  payment.events

    - Количество партиций на регион: 2
    - Ключ партиционирования: region:saga<sub>id</sub>
    - Retention: 30 дней (аудит)
    - Consumer capacity: 100 RPS
    - Обоснование: аналогично order.events

5.  analytics.events

    - Количество партиций: 3
    - Ключ партиционирования: - (для простой записи в ClickHouse)
    - Retention: 90 дней (ML обучение)
    - Consumer capacity: не ограничивает
    - Обоснование: аналитика не критична к задержкам, можно использовать меньше партиций

6.  dead.letter.queue

    - Количество партиций: 1
    - Ключ партиционирования: нет
    - Retention: 7 дней
    - Обоснование: очередь необработанных событий, не требует параллельной обработки

### Региональная изоляция

Региональная изоляция обеспечивается разными кластерами, а не партициями.

- В каждом регионе (Asia, SA, Europe) развёрнут свой независимый кластер брокера.
- Партиции (2, 7, 3) — это конфигурация внутри одного регионального кластера.
- Ключ region:saga<sub>id</sub> нужен для локальной маршрутизации внутри региона, а не для отправки в другой регион.

## Аргументация

Расчет нагрузки на масштабирование следующий

### Исходные данные

|  |  |
|----|----|
| Показатель | Значение |
| ——————————————– | ——————————- |
| Конкурентных поездок (одновременно активных) | 500 000 |
| Регионов | 3 (Asia, South America, Europe) |

Количество партиций определяется индивидуально для каждого топика, исходя из его RPS и пропускной способности consumer'a.

### Заказы

1.  Допущения

    - Средняя длительность поездки 15 минут
    - Нагрузка распределяется равномерно по регионам
    - Пропускная способность consumer до 100 RPS (реалистичная оценка для Python/Java с не очень тяжёлой логикой).

2.  Расчет

    Оборот заказов в час = 500 000 × (60 мин / 15 мин) = 2 000 000 заказов в час Заказов в секунду = 2 000 000 / 3600 = 556 RPS Нагрузка на регион = 556 / 3 ≈ 185 RPS на регион Партиций на регион = 185 / 100 ≈ 1.85 ≈ 2 партиции

3.  Результат

    Для 200 RPS на регион достаточно 2 партиций

4.  Когда увеличивать

    Когда нагрузка на регион превысит

    - 200 RPS (100 RPS × 2 партиции = предельная пропускная способность).
    - это ~540 000 конкурентных поездок

### Водители

1.  Допущения

    - Водителей онлайн: 2 на один заказ
    - Интервал отправки геопозиции: 5 сек
    - Пропускная способность до 10 000 RPS - лёгкая обработка, только запись в Elasticsearch

2.  Расчет

    Нагрузка на регион: 1 000 000 / 5 сек / 3 ~67 000 RPS на регион Партиций на регион: 67 000 / 10 000 = 6,7 ≈ 7 партиции

### Почему разное количество партиций для разных топиков

- order.events: 185 RPS, лёгкая нагрузка → 2 партиции достаточно
- driver.location: 67 000 RPS, высокочастотный поток → 7 партиций
- analytics.events: не критично → 3 партиции

### Почему композитный ключ \`region:saga<sub>id</sub>\`

- Префикс \`region\` позволяет (при необходимости) реализовать кастомный партиционер для локальной обработки
- Полная строка гарантирует, что все события заказа попадают в одну партицию
- Без префикса регион не влиял бы на распределение (хеш от saga<sub>id</sub> одинаков везде)

## Ограничения и риски

|  |  |  |
|----|----|----|
| Риск | Вероятность | Смягчение |
| —————————————————————- | ———– | —————————————————————— |
| ****Неравномерное распределение нагрузки по партициям**** | Средняя | Мониторинг размера партиций, ручная балансировка при необходимости |
| ****Рост числа партиций снижает производительность брокера**** | Низкая | 7 партиций на регион |
| ****Переход на новый топик при масштабировании требует остановки**** | Низкая | Стратегия "новый топик" с переключением consumer'ов без простоя |
| ****Оценка числа водителей (2 на заказ) может быть неточной**** | Средняя | Партиции driver.location заложены с запасом (7 вместо 6.7) |

## Альтернативы

|  |  |
|----|----|
| Альтернатива | Почему отклонена |
| ———————————- | ————————————————————— |
| ****Увеличение партиций на месте**** | Ломает порядок событий одного заказа (перераспределение ключей) |
| ****Один топик на все типы событий**** | Сложно управлять, разные retention, разные ключи |
| ****Одна партиция на топик**** | Нет горизонтального масштабирования, узкое место |
