# ADR-<u>\_</u>: Выбрать технологию для динамического ценообразования

## Контекст

****Подход к динамическому ценообразованию:**** Гибридный.

- Базовая цена рассчитывается раз в 5–10 секунд на основе потоковых данных (PyFlink)
- Результат кладётся в Redis
- Синхронный вызов \`pricing.calculate()\` читает из Redis (50 мс)

****Почему не полностью потоковый:**** Слишком высокая задержка для UX такси (200–500 мс).

****Почему не чистое предвычисление:**** Не отражает резкие изменения спроса (концерт, дождь).

## Проблема

В динамическом ценообразовании (surge pricing) нужно:

- Агрегировать события за последние N секунд (количество заказов, свободные водители)
- Обрабатывать события по времени их реального наступления (event time), а не по времени получения (processing time)
- Учитывать опоздавшие события (late data) — водитель отправил геопозицию с задержкой из-за плохой сети

Ключевая проблема:

- если водитель прислал геопозицию с задержкой, мы должны использовать время её отправки (когда он был в этой точке), а не время получения.

## Решение

Для динамического ценообразования и других задач тяжелой аналитики выбран PyFlink + H3 + Side outputs

- потоки расчет аналитики
  - `analytics-aggregator`: оконные агрегации (1 мин, 1 час) в ClickHouse (`analytics.events`)
  - `fraud-detector`: детектор аномалий (окна, сложные паттерны) (`payment.events`)
- Геохеширование (H3 в PyFlink)
  - чтобы считать цену не «вообще», а в конкретном квадрате города.
- Side Outputs
  - основная выдача идет в базу, а оперативная — в шину

****Как это работает:****

1.  Flink-джоба читает `order.events` и `driver.location` из RabbitMQ Streams
2.  Вычисляет соотношение заказов к свободным водителям по регионам
3.  Результат (коэффициент спроса) пишется в `pricing.events`
4.  Pricing Service читает \`SurgeFactorCalculated\` и кеширует в Redis для синхронных запросов от пассажиров

## Аргументация

1.  Гибридная производительность (C++ под капотом, Python в интерфейсе)
2.  Event Time и «Опоздавшие данные» — из коробки
3.  Интеграция с ML-экосистемой Python
4.  Унифицированный SQL (Low Code для аналитиков)

Выбрав PyFlink, мы

- даем Python-команде инструмент на уровне индустриального стандарта, который выдержит нагрузку Uber,
- но при этом сохраняем высокую скорость разработки на Python.
  - не заставляем команду учить Java
  - не тратим время на борьбу с Garbage Collector в Java или написание своих оконных функций на Python

## Ограничения и риски

Настройка окружения PyFlink (Python на воркерах Flink) требует навыков DevOps.

Описывай гео-хеширование (H3) через SQL User-Defined Function (UDF) на Java (один раз написать/скачать готовую), а вызывай её из PyFlink. Так как

- PyFlink наиболее эффективен, когда вы используете Table API или SQL
  - так как в этом случае данные не покидают JVM (обработка идет на C++/Java), что критично для нагрузки в 500к поездок.
- Если писать чистый Python-код внутри функций Flink (UDF), то производительность упадет из-за накладных расходов на сериализацию между Python и Java.

## Альтернативы

****Почему не Kafka Streams или Faust Streaming:****

- Динамическое ценообразование требует обработки по event time (время реального наступления события, а не время получения)
- Необходима точная работа с опоздавшими данными (late data) от водителей
- Сложные оконные агрегации (скользящие окна 5 и 15 минут) удобнее реализовывать во Flink \[citation:1\]

****Почему не делать расчёт внутри Pricing Service:****

- Pricing Service отвечает синхронно (50 мс), расчёт коэффициента — тяжёлая вычислительная задача
- Вынос расчет вовне позволяет масштабировать обработку независимо

# Потребители событий

В системе используются 3 типа потребителей:

- Обычные consumer'ы (Faust Streaming)
  - `notif`: читает `order.events`, отправляет push
  - `booking`: читает `command.assignDriver`, обновляет статус
