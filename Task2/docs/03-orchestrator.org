** Контекст
При переходе на микросервисную архитектуру ACID-транзакции внутри одного монолита заменяются распределёнными транзакциями между сервисами. Для обеспечения согласованности данных (например, "платёж прошёл → заказ создан → водитель назначен") необходим паттерн Saga.

** Проблема
- При выделении первого бизнес-сервиса (Payments) ACID-транзакции ломаются
- При отказе одного сервиса другие должны откатить свои изменения (компенсации)
- Необходимо отслеживать состояние долгих операций (поиск водителя до 30 секунд)
- Клиент должен получать статус обработки заказа

** Решение
Выбран паттерн **Saga Orchestration** (оркестрация), где центральный координатор управляет последовательностью шагов и компенсациями.
* Оркестратор
Temporal Python SDK

** Функции
1. Хранит состояние саги 
1. Вызывает сервисы синхронно
1. Обрабатывает таймауты
1. Вызывает компенсации       
1. Отвечает клиенту

** Взаимодействует синхронно
1. Проверить фрод
1. Заблокировать деньги
1. Запустить обработку
1. Отчитаться о принятии заказа

** Взаимодействует асинхронно
1. отслеживать статус заказа
1. отменить заказ и вернуть деньги, если заказ никто не принял
1. завершить заказ и провести списание денег, когда заказ исполнен
1. инициировать оповещения участникам заказа

** Аргументация
**Оркестратор отдельно от BookingService**
| Аргумент                                               | Обоснование                                                                           |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------- |
| **Разделение ответственности (Single Responsibility)** | BookingService хранит статус заказа, Оркестратор управляет сагой                      |
| **Независимое масштабирование**                        | При росте заказов масштабируем оркестратор отдельно                                   |
| **Логика саги в одном месте**                          | Изменение последовательности шагов не требует правки BookingService                   |
| **Устойчивость к отказам**                             | Temporal переживает падения оркестратора, BookingService остаётся доступен для чтения |

*** Почему синхронные вызовы для fraud и payments.hold

- Пользователь ждёт подтверждения: «Заказ принят, деньги заблокированы»
- Fraud проверка быстрая (100 мс), холд платежа — до 3 секунд (приемлемо)
- Асинхронная обработка этих шагов ухудшит UX (пользователь не видит результат)

*** Почему асинхронные для остального

- Поиск водителя может длиться до 30 секунд
- Уведомления и списания не требуют немедленного ответа клиенту
- Temporal отлично справляется с долгими workflow

** Ограничения и риски
| Риск                                           | Вероятность | Смягчение                                                       |
| ---------------------------------------------- | ----------- | --------------------------------------------------------------- |
| **Temporal — дополнительная инфраструктура**   | Средняя     | Использовать Temporal Cloud (managed) или поднять кластер в K8s |
| **Сложность отладки workflow**                 | Низкая      | Temporal UI предоставляет визуализацию саг, логирование         |
| **Задержка при вызове сервисов**               | Низкая      | Таймауты настроены, retry policies                              |
| **Потеря состояния саги при падении Temporal** | Низкая      | Temporal использует persistent storage (Cassandra / PostgreSQL) |
| **Синхронные вызовы могут тормозить**          | Средняя     | Мониторинг длительности саг, алерт при >5 сек                   |

** Альтернативы
| Альтернатива                                  | Почему отклонена                                                                   |
| --------------------------------------------- | ---------------------------------------------------------------------------------- |
| **Хореография (Choreography)**                | Логика размазывается по сервисам, сложно отлаживать, риск циклических зависимостей |
| **Ручная реализация Saga на Django + Celery** | Нет встроенных компенсаций, таймаутов, наблюдаемости                               |
| **Kafka Streams + State Store**               | Очень сложно, требует ручной реализации компенсаций                                |
| **AWS Step Functions**                        | Вендор-лок, плохая локальная разработка, Java/JS SDK                               |
| **SAGA pattern с Atomikos (JTA)**             | Требует XA-совместимых БД, не масштабируется                                       |
