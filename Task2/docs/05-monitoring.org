** Контекст
Система GoFuture переходит на микросервисную архитектуру с тремя географическими регионами (Asia, South America, Europe).

Требования к наблюдаемости:
- Обеспечить 99.99% доступности критических сервисов
- Сократить MTTR (среднее время восстановления) с 4+ часов до 15 минут
- Обеспечить возможность отладки распределённых транзакций (саг) по =saga_id=
- Соблюдать требования локальных регуляторов к хранению данных

** Проблема
В распределённой системе с 10+ микросервисами и асинхронными событиями:

1. **Сложно найти причину сбоя** — ошибка может возникнуть в любом из 10 сервисов
2. **Невозможно отследить путь запроса** — запрос проходит через оркестратор, fraud, payments, driver
3. **Метрики, логи и трейсы не связаны** — нельзя перейти из упавшей метрики в логи ошибки
4. **Разные команды используют разные инструменты** — усложняется поддержка

** Решение
Выбран подход =Metrics + Alerts + Logs + Tracing с разделением на технический и бизнес-уровни.

Диаграмма - Observability
*** Инструменты
Для реализации сбора и обработки телеметрии выбран стек Grafana
- Grafana Alloy - сбор и обогащение метрик логов и трейсов
- Prometheus - сбор метрик
- Prometheus Alert Manager - управление системными уведомлениями
- Loki - сбор и хранение логов
- Tempo - сбор и хранение трейсов
- Grafana - визуализация

Код инструментируется:
- prometheus_client официальная python библиотека для генерации метрик
- otel auto-instrumentation библиотека автогенерации трейсов
*** Технический мониторинг (SRE)
Что отслеживаем:
- Доступность сервисов (uptime)
- Задержки (latency p50, p95, p99)
- Ошибки (error rate, exceptions)
- Насыщение (CPU, memory, network, connections)
- Трафик

Для кого: SRE, дежурные инженеры
Пример дашборда: «Золотые сигналы» (Google SRE book) — задержка, трафик, ошибки, насыщение.

*** Бизнес-мониторинг (продуктовые метрики)
Что отслеживаем:
- Количество заказов в минуту/час
- Процент успешно завершённых саг
- Средний ETA
- Коэффициент отмен (по причинам: фрод, нет водителя, таймаут)

Для кого: Продуктовые аналитики, менеджеры
Пример дашборда: «Заказы по регионам, успешность саги, среднее время поиска водителя».

*** Логи (структурированные)
- Структурированные (JSON), Loki + Otel Grafana Alloy
- Обязательные поля:
    - saga_id,
    - event_id,
    - event_type,
    - service,
    - region

Для кого: SRE, разработчики

*** Распределённая трассировка
- Tail Sampling - условия сохранения трейсов: ошибка, задержка, атрибут 
- Пакетирование
- Обогащения

Цели:
- Оркестратор
    - Весь процесс создания заказа (Saga) от начала до конца.
- Сервисы:
    - Входящие HTTP-запросы (от API Gateway), исходящие вызовы к другим сервисам (синхронные), запросы к базам данных (PostgreSQL, Redis), вызовы внешних API (Яндекс.Пэй, Яндекс.Карты).

Для кого: SRE, разработчики (отладка по saga_id)

** Аргументация
** Почему Grafana Alloy?**

1. **Универсальность** — соответствует стандарту OpenTelemetry
2. **Единый агент** — собирает логи, метрики и трейсы в одном месте
3. **Безопасность** — упрощает настройку сетевых политик между регионами
4. **Push-модель** — позволяет собирать данные с короткоживущих задач
5. **Сбор метрик инфраструктуры** — серверы, Docker, Kubernetes
6. **Актуальность** — официальная замена устаревшим Promtail и Grafana Agent

**Почему Grafana Tempo?**
1. **Масштабируемость** — линейно масштабируется с ростом объёма трейсов
2. **Экономичность** — не требует индексации, хранит трейсы в объектном хранилище
3. **Интеграция** — нативная работа с Grafana, Loki, Prometheus
4. **Correlation** — бесшовный переход из метрики → лог → трейс

**Почему OpenTelemetry Auto-Instrumentation?**
1. **Без изменения кода** — достаточно добавить флаг при запуске
2. **Единый стандарт** — не привязан к конкретному вендору
3. **Поддержка Python** — официальная поддержка Django, Flask, FastAPI

** Ограничения и риски
| Риск | Вероятность | Смягчение |
|------+-------------+-----------|
| **Перегрузка Alloy** | Средняя | Настроен =memory_limiter= и =batch= |
| **Потеря трейсов при высокой нагрузке** | Низкая | Tail sampling + горизонтальное масштабирование Alloy |
| **Рост стоимости хранения трейсов** | Средняя | Tail sampling (экономия 90-95%) |
| **Задержки в доставке логов** | Низкая | Loki принимает push, Alloy буферизирует |
| **Сложность отладки распределённых саг** | Низкая | Решено через =saga_id= и correlation |

** Альтернативы
| Альтернатива | Почему отклонена |
|--------------+------------------|
| **Promtail + Grafana Agent (раздельно)** | Устаревшие инструменты, поддержка прекращается в 2026 году |
| **ELK Stack (Elasticsearch, Logstash, Kibana)** | Тяжёлый, ресурсоёмкий, нет нативной интеграции с Prometheus |
| **Jaeger + All-in-one** | Tempo лучше интегрируется с Grafana Stack |
| **Datadog / New Relic** | Проприетарные, дорогие, данные покидают регион (комплаенс) |
