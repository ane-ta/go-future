| **Критерий**                   | **Анализ на основе предоставленных данных**                                                                                                                                                                                                                                                                  |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Тип и Свойства процесса**    | Смешанный тип:<br>- **OLTP-транзакции** (бронирование поездок) , <br>- **Real-time стриминг** (геолокация водителей, динамические цены) ,<br>- **аналитика** (ClickHouse, Spark/Flink).<br>- Процесс содержит сложные ветвления и высокую степень асинхронности (Celery, RabbitMQ).                          |
| **Масштаб и Нагрузка (Scale)** | **Highload**. <br>- Переход от 100 тыс. к 500 тыс. конкурентных поездок. <br>- Объем данных — **Тб** (учитывая логи, историю поездок и ML-пайплайны).<br>- Требования к Latency — критически низкие для матчинга водителя и пассажира .                                                                      |
| **Критичность данных**         | **Availability > Consistency** (согласно теореме CAP).<br>- Для системы заказа такси доступность (возможность уехать) критичнее мгновенной согласованности баланса во всех регионах.<br>- Однако <br>  - для платежей требуется **ACID**, <br>  - а для статусов поездки допустима **Eventual Consistency**. |
| **Сложность логики**           | **«Ювелирная» обработка**. <br>- динамическое ценообразование в реальном времени ,<br>- «умное» перераспределение водителей (предотвращение «горячих зон»)<br>- ML-антифрод.                                                                                                                                 |
| **Жизненный цикл и Состояние** | **Stateful**.<br>- необходимо помнить состояние каждой активной поездки (Restartability)<br>- и хранить историю для аналитики и регуляторов .                                                                                                                                                                |
| **Инфраструктурный контекст**  | Переход от **Monolith (Django/Python)** к **Microservices**. <br>Облако: **Yandex Cloud** (указан как Cloud Provider). <br>Текущая контейнеризация — Docker.                                                                                                                                                 |

4 технологических блока:

1. **Decomposition & Integration:** 
    для снижения связности
    - Распил монолита на домены (Task 1) 
    - и внедрение Event-Driven Communication (Task 2).
    
2. **Global Infrastructure:** 
    для обеспечения доступности 99,95%
    - Настройка Geo-distribution
    - и Failover в >3 регионах  (Task 3).
    
3. **Data & Intelligence:** 
    - Построение пайплайна сбора данных для ML-моделей прогнозирования спроса (Task 4).
    
4. **Platformization:** 
    - Реализация мультитенантности 
    - и IAM для безопасного подключения партнеров (Task 5).
