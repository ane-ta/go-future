# План миграции данных

## Принципы

1.  Zero downtime — данные мигрируются без остановки сервисов
2.  Обратимость — возможность отката на каждом этапе
3.  Идемпотентность — повторная миграция не ломает данные

## Общий подход для каждого сервиса

| Этап | Действие | Инструменты |
|----|----|----|
| 1\. Подготовка | Создаём новую БД сервиса, настраиваем репликацию | PostgreSQL, логическая репликация |
| 2\. Начальная синхронизация | Копируем существующие данные из монолитной БД | pg<sub>dump</sub> / COPY, Debezium snapshot |
| 3\. CDC | Настраиваем поток изменений из монолитной БД в новую | Debezium + RabbitMQ Streams |
| 4\. Dual-write период | Новый сервис пишет в обе БД (свою и монолитную) | Временный код в сервисе или CDC |
| 5\. Переключение чтения | Запросы на чтение идут в новую БД | Feature flags, API Gateway |
| 6\. Переключение записи | Запросы на запись идут только в новую БД | Отключаем dual-write |
| 7\. Очистка | Удаляем дублирующиеся данные из монолитной БД | Скрипты очистки, архивация |

## Пример для сервиса Payments

| Шаг | Действие | Время | Риск |
|----|----|----|----|
| 1 | Создаём payments<sub>db</sub> (PostgreSQL) | 1 час | Низкий |
| 2 | Начальный экспорт таблицы payments из монолита | 2 часа (на 100GB) | Низкий |
| 3 | Настраиваем Debezium CDC из монолита в payments<sub>db</sub> | 1 день | Средний |
| 4 | Запускаем Payments Service в dual-write режиме | 1 неделя | Средний |
| 5 | Переключаем чтение через feature flag (1% → 100%) | 1 день | Низкий |
| 6 | Отключаем запись в монолитную БД | 1 час | Низкий |
| 7 | Удаляем дубли payments из монолитной БД | 1 день | Низкий |

## Особенности для сервисов с большим объёмом данных

| Сервис | Объём данных | Особенность миграции |
|----|----|----|
| Analytics | Самый большой (все данные) | Мигрируется последним, до миграции — CDC в монолитную БД |
| Geo | Большой (Elasticsearch индексы) | Требует reindex при изменении маппинга |
| Fraud | Большой (история) | Можно мигрировать только последние N месяцев |

## Механизм отката

Если на шаге 5 или 6 обнаружена проблема:

1.  Выключаем feature flag (чтение возвращается на монолитную БД)
2.  CDC продолжает синхронизировать данные
3.  Исправляем ошибку, повторяем шаги 5-6

Потеря данных: `нет`, монолитная БД остаётся источником правды до полного переключения.

## После завершения миграции всех сервисов

1.  Аналитика переключается с монолитной БД на Kafka (один раз)
2.  Монолитная БД больше не нужна для операций
3.  Архивация и удаление дублирующихся данных
4.  Отключение CDC-коннекторов
