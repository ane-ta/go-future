# ADR-001: Миграция монолита на микросервисы

## Контекст

### Бизнес-цели (SLO)

- 500 тыс. конкурентных поездок без каскадных сбоев
- Время сборки и деплоя \< 15 минут
- Время выхода новой функции \< 2 недель

### Проблемы текущей архитектуры

- Каскадные отказы: падение Pricing валит весь заказ
- Долгая сборка: \> 30 минут
- Медленный выпуск функций: до 6 месяцев

## Проблема

Спроектировать архитектуру, соответствующую нефункциональным требованиям. Описать процесс миграции, минимизирующий время простоя системы.

## Решение

Выбрана стратегия эволюционной миграции (Strangler Fig Pattern) - постепенное выделение сервисов с обеспечением обратной совместимости на промежуточных этапах. Является противоположностью стратегии большого взрыва (Big Bang) - переписывания всей системы разом.

Выделяемые микросервисы - диаграмма Обеспечение обратной совместимости - диаграмма Async среда - RabbitMQ Streams

## Аргументация

### ➕ Минимизация времени простоя

- **Можно:**
  - Не выключать ни один сервис во время миграции
- **Так как:**
  - API Gateway + Strangler Fig позволяют переключать трафик постепенно (1% → 100%)
  - Feature flags дают мгновенный откат при ошибках
  - Token exchange (session<sub>id</sub> → JWT) позволяет старому монолиту и новым сервисам работать параллельно

### ➕ Быстрый прирост производительности и масштабируемости

- **Можно:**
  - Выдерживать возрастающую нагрузку
- **Так как:**
  - В первой волне миграции выносятся самые ресурсоемкие сервисы
    - Fraud разгружает БД от конфликта тяжёлых SELECT и частых INSERT
    - Payments разгружает пул от соединений с платежной системой
    - Geo разгружает монолит высокочастотными чтениями и записями перемещений водителей
  - Становится доступным оперативное использования паттернов надежности при временных и постоянных сбоях.

### ➕ Минимизация рисков

- **Можно:**
  - Рассчитывать на минимальное количество инцидентов в процессе миграций
- **Так как:**
  - Процесс миграции отрабатывает на сервисах с низкой сложностью и низкой критичностью
  - Сложность сервисов наращивается постепенно
  - Для среды общения используется RabbitMQ Streams, который добавляется к уже развернутому брокеру.
    - Это позволяет реализовать replay и партиционирование без внедрения нового брокера. При росте нагрузки до 1M+ RPS или появлении требований к сложной stream-обработке, возможно миграция на Kafka.»

### ➕ Согласованность данных

- **Можно:**
  - Гарантировать согласованность транзакций в процессе миграции
- **Так как:**
  - Оркестратор внедряется до миграции этапов обработки заказов

## Ограничения и риски

### ⚙️ Дублирование данных (х2-х3)

- **Нужно:**
  - Выделить ресурсы под хранилища в размере х2-х3 на период миграции
- **Так как:**
  - Все новые сервисы пишут и в свою БД, и в монолитную (через CDC) для поддержания работы Analytics
  - К концу миграции монолитная БД содержит дубли всех данных
- **Решение:**
  - Можно выделить Analytics на первых этапах, продолжая читать данные из базы монолита
  - По мере выделения сервисов переписывать Analytics для перехода на async публикацию аналитики каждым сервисом самостоятельно

### ⚙️ Временная инфраструктурная сложность

- **Нужно:**
  - Натсраивать CDC (Debezium), API Gateway exchage token
  - Писать промежуточный вариант оркестратора
- **Так как:**
  - Требуется обеспечить обратную совместимость при минимальном изменении кода монолита.

### ⚙️ Задержка репликации

- **Нужно:**
  - Учитывать, что монолит видит данные выделенных сервисов не в реальном времени
- **Так как:**
  - Согласованность данных достигается через CDC, который работает асинхронно
- **Решение**
  - Для бизнес-транзакций (заказы, платежи) критичные данные не проходят через CDC — они идут через оркестратор синхронно
  - CDC используется только для аналитики и некритичных к задержкам сценариев
  - При проектировании учитывать, что аналитика может быть слегка устаревшей (eventual consistency)

### ⚠️ Риск рассогласования данных

- **Когда:**
  - При настройке CDC между сервисами и монолитом
- **Может:**
  - Произойти рассогласование данных при ошибки в конфигурировании.
- **Так как:**
  - CDC зависит от правильной настройки:
    - Логическая репликация в PostgreSQL (wal<sub>level</sub>, publication, slot)
    - Конфигурация Debezium (начальная синхронизация, партиционирование, обработка ошибок)
    - Схема данных (изменение структуры таблицы может сломать парсинг CDC)
- **Решение:**
  1.  Мониторинг CDC — отслеживать лаг репликации (Lag), ошибки коннекторов, dead letter queue
  2.  Идемпотентность — потребители CDC должны быть готовы к дублированию событий
  3.  Точка восстановления — хранить offset в Kafka для возможности перезапуска с последней успешной позиции
  4.  Тестирование — развернуть staging-окружение с полным циклом CDC перед продакшеном
  5.  Fallback-сценарий — при падении CDC аналитика временно переключается на чтение из БД сервисов напрямую (через специальный адаптер)

### ⚠️ Риск использования RabbitMQ вместо Kafka

- **Когда:**
  - Нагрузка возрастет
- **Может:**
  - Упереться а ограничения RabbitMQ Streams
- **Так как:**
  - Операции восстановления после падения узла могут быть дольше, чем у Kafka
  - Нет нативной поддержки сложных stream-processing (Flink, Spark Streaming)
  - Инструменты мониторинга беднее
- **Решение:**
  - При нагрузке до 500k поездок — достаточно
  - При росте до 1M+ RPS или появлении требований к сложной аналитике — миграция на Kafka (заложено в архитектуру)
