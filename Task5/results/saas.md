# ADR-\_: Мультитенантность

## Контекст

«GoFuture» стремится быстро запускать решения для партнёров в новых регионах с полной изоляцией данных и кастомизацией функциональности.

## Проблема

Разработать дизайн мультитенантной платформы

- модель изоляции
- IAM-систему с ролевым доступом и SSO
- автоматизированный процесс подключения новых партнёров (onboarding)
- систему мониторинга мультитенантного окружения

## Решение

- модели изоляции: Schema-per-Tenant
  - В одной базе данных (PostgreSQL) мы создаем для каждого партнера (арендатора) отдельную схему (tenant<sub>A</sub>.orders, tenant<sub>B</sub>.orders)
- Стратегия идентификации: Header-based identification
  - API Gateway извлекает tenant<sub>id</sub> из JWT и помещает в заголовок
  - Микросервисы используют заголовок, чтобы выставить схему бд как контекст последующих SQL запросов (Dynamic DataSource Router)
  - Микросервисы публикуют в событиях tenant<sub>id</sub>
  - Monitoring System поддерживает тег tenant<sub>id</sub> во всех метриках как label.
- Стратегия управления доступом (IAM & SSO) IdentityServer + Realm
  - IdentityServer: Keycloak (Open Source стандарт).
  - Realm (Область): Для каждого партнера создается свой Realm. Это как отдельная база пользователей.
  - Сотрудники тенанта (водители)
    - регистрируются по входному урлу уникальному для кажого тенанта.
    - логинится, и в их JWT-токене (паспорте) уже содержится tenant<sub>id</sub>: 123.

### Onboarding

Процесс «заселения» нового партнера (сверху вниз):

1.  Бизнес-уровень: Менеджер GoFuture вводит данные нового партнера (название, регион).
2.  Инфраструктурный уровень (Оркестратор делает работу):
    1.  Идет в IAM → Создает новый Realm и учетку Partner Admin.
    2.  Идет в Database Manager → Выполняет SQL: CREATE SCHEMA tenant<sub>ID</sub>. Накатывает туда таблицы.
    3.  Идет в Monitoring Configurator → Создает тег (Label) для этого партнера в Grafana/Prometheus, чтобы фильтровать метрики.
3.  Готовность: Партнер получает ссылку на свою админку

### Матрица ролей и уровней доступа (RBAC)

|  |  |  |  |
|----|----|----|----|
| Роль | Область видимости (Data Scope) | Уровень доступа к данным | Описание ключевых функций |
| —————– | ———————————– | ————————- | ————————————————————————————————————– |
| ****Global Admin**** | Вся платформа (все партнеры) | ****Полный (RW)**** | Создание и удаление партнеров, управление глобальным биллингом, доступ к общей аналитике. |
| ****Partner Admin**** | Только свой партнер (Tenant Schema) | ****Административный (RW)**** | Управление сотрудниками своего таксопарка, настройка локальных тарифов и правил. |
| ****Dispatcher**** | Только свой партнер (Tenant Schema) | ****Операционный (RW)**** | Работа с заказами, назначение водителей, просмотр истории поездок своего таксопарка. |
| ****Driver**** | Свои данные внутри партнера | ****Ограниченный (R/W)**** | Доступ только к своим заказам, статусу поездки и финансовой статистике. Нет доступа к данным других водителей. |
| ****Support**** | Свой партнер (Tenant Schema) | ****Только чтение (R)**** | Решение спорных ситуаций, просмотр логов и деталей поездок без права изменения фин. настроек. |

## Аргументация

### Почему Schema-per-Tenant

1.  Баланс цены и качества:
    - сервер один, но данные разделены.
2.  Безопасность (No Cross-Tenant leaks):
    - Труднее смешать разные схемы в SQL-запросе, чем потерять идентификатор тенанта.
3.  Легкий бэкап и миграция:
    - Если один партнер уходит, ты можешь удалить или выгрузить только его схему, не трогая остальных.
4.  Гибкость:
    - Можно обновлять структуру таблиц у одного партнера (например, для кастомизации под законы Индонезии) чуть раньше, чем у других.

### Почему Onboarding service

Позволяет разворачивать инфраструктуру для нового партнера без участия DevOps-инженеров.

### Почему Dynamic DataSource Router

Минимизирует накладные расходы на переключение контекста, позволяя использовать общий пул соединений.

## Ограничения и риски

Общие ресурсы (Noisy Neighbor):

- Один партнер может перегрузить общую БД.
- Митигация:
  - Внедрение квот и мониторинга в разрезе tenant<sub>id</sub>

## Альтернативы

### Почему не Shared Schema

Если бы мы использовали одну таблицу с колонкой tenant<sub>id</sub>, риск утечки данных из-за ошибки в коде (забытый WHERE) был бы слишком велик для B2B-платформы.
